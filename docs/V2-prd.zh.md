# Rhythm V2 PRD（Fork 草案）

## 1. 文档定位

本文档用于记录我在本地 fork 上继续推进的产品方向与近期待办，不替代上游 V1 设计文档。

- `docs/V1-design.md` 继续作为上游 `main` 分支的 V1 基线与历史设计记录
- 本文档描述 fork 中已经探索过的交互差异，以及接下来可能推进的 V2 方向

换句话说：如果要了解当前上游 V1 是什么，看 `docs/V1-design.md`；如果要了解这个 fork 准备往哪里走，看本文档。

## 2. 当前基线与已探索差异

### 2.1 上游 V1 基线

当前上游 `main` 仍然以稳定可用的节奏提醒工具为目标，核心能力包括：

- 菜单栏常驻应用
- 专注 / 休息节奏配置
- 全屏休息遮罩
- `ESC` 跳过休息
- 锁屏后重置周期
- 不休息模式
- 本地 session 记录

### 2.2 Fork 已落地的行为变化

本地 fork 现在已经在上游 V1 基线之外落地了以下行为：

1. 当前专注阶段可直接调整当前 phase
   - `延长专注 5 分钟`
   - `延长专注 10 分钟`
   - `提前休息 5 分钟`
2. `提前休息 5 分钟` 只在至少还剩 5 分钟时可用
3. 当前休息阶段可直接延长当前 phase
   - `延长休息 1 分钟`
   - `延长休息 5 分钟`
4. 默认设置与当前 phase 的语义被拆开
   - 修改 `专注间隔` 与 `休息时长` 时，影响下一轮周期
   - 当前 phase 的增减时长，只影响当前 phase
5. 菜单栏可见性与相关 UI 稳定性经过额外加固
   - 包括菜单栏图标可见性优化
   - 菜单栏中原本显示的 `Rhythm` 文案已改为保留图标并展示实时倒计时
   - 包括状态项被系统移除后的恢复能力
6. fork 现在已经支持中英双语界面
   - 菜单栏设置中可以切换 `中文` 和 `English`
   - 首次使用时，只有 `zh*` 系统语言默认中文，其它系统语言默认英文
   - 后续新增的用户可见功能默认应继续保持双语
7. 更长休息能力已经成为当前 fork 的已发布基线
   - 普通休息默认时长现在最高可设到 20 分钟
   - 菜单中提供 `用餐`、`健身`、`小憩`、`外出`、`桌前休息` 预设
   - `桌前休息` 明确是非锁屏、可继续使用电脑的休息类型
   - 普通休息和离开屏幕型长休息仍保留全屏遮罩
   - session 记录现在会保存休息类型，以及计划时长和实际时长

## 3. V2 想解决的问题

V1 已经能完成“提醒休息”这件事，但 fork 想进一步解决的是：当用户正在使用应用时，是否能更自然地微调当前节奏，而不必被迫重置整个周期。

V2 的核心目标：

1. 让“默认节奏”和“当前 phase 调整”成为两套清晰、不互相打断的规则
2. 让常见临时决策可以在菜单栏或遮罩上直接完成，而不是通过重置计时曲线来实现
3. 把双语界面作为后续功能的默认基线，并在不引入复杂系统的前提下继续为更长休息场景留出空间

## 4. V2 产品方向

### 4.1 Phase 级临时控制

保留并整理 fork 中已经出现的 phase 控制模型：

- 专注阶段支持：
  - 提前休息 5 分钟
  - 延长专注 5 分钟
  - 延长专注 10 分钟
- 休息阶段支持：
  - 延长休息 1 分钟
  - 延长休息 5 分钟

该模型的设计原则：

- 默认设置代表“下一轮应该怎样开始”
- phase 控制代表“这一轮现在怎么调整”
- 两类操作不能互相覆盖语义

### 4.2 设置生效规则

V2 倾向采用以下规则作为正式语义：

1. 用户修改 `专注间隔` 时，不重置当前专注阶段
2. 用户修改 `休息时长` 时，不改变已经开始的休息阶段
3. 新设置在下一轮开始时生效
4. 手动延长 / 缩短 phase 只作用于当前轮次

这样做的好处是：

- 避免用户在调默认值时意外打断当前节奏
- 避免“修改设置”和“临时补时”混成同一种操作
- 为后续引入更多快捷控制打下统一规则

### 4.3 菜单栏稳定性

fork 会继续把菜单栏存在感和可恢复性视为基础体验，而不是额外 polish。

短期内默认认为以下目标成立：

- 菜单栏图标在浅色/深色环境下都应足够清晰
- 菜单栏状态项在专注和休息阶段都应显示当前倒计时，并尽量避免宽度抖动
- 状态项不应因为系统或 UI 状态异常而悄悄消失
- 即使出现异常，也应优先恢复菜单栏入口，而不是要求用户重启应用

## 5. 近期方向

以下内容构成 V2 近期路线，其中有些已经成为当前基线，有些仍然是待明确的产品方向：

### 5.1 双语界面基线

fork 现在把双语界面视为已落地的基础能力，而不是未来候选项。

当前基线包括：

- 可见应用界面同时支持中文和英文
- 菜单栏面板中提供 `中文` / `English` 语言切换
- 首次使用时采用简单规则：`zh*` 系统语言使用中文，其它系统语言使用英文
- 后续新增的用户可见功能应直接提供中英双语，而不是先做单语版本

### 5.2 更长休息能力基线

更长休息在当前 fork 中已经不是纯候选方向，而是已落地能力。当前模型是：

- 普通短休息仍然保留，并且默认时长现在最高可设到 20 分钟
- `用餐`、`健身`、`小憩`、`外出` 属于更长的离屏休息，继续使用全屏遮罩
- `桌前休息` 是明确的“还在电脑前，但不做工作”的休息类型，适合看视频、刷动态等场景
- `桌前休息` 结束时，Rhythm 会自动恢复专注，并在系统允许通知时提醒用户

这样做的目的是：只保留一个清晰的“可继续用电脑”的休息类别，而不是让每种长休息都变成独立行为模型。

### 5.3 每日专注 / 休息总量

在不引入复杂图表系统的前提下，补上轻量统计，帮助用户回答两个简单问题：

- 今天大概专注了多久
- 今天大概休息了多久

### 5.4 重新审视锁屏重置

V1 里“锁屏即重置”足够简单，但在更灵活的 phase 语义下，是否还应保持完全重置，需要重新评估。

可能需要确认：

- 锁屏后是否一定回到完整专注周期
- 锁屏前已经做过的 phase 延长是否应丢弃
- 短暂锁屏与长时间离开是否应采用同一策略

## 6. 非目标

当前 V2 草案仍然不准备立即引入：

- 云同步
- 多设备同步
- 复杂报表或图表系统
- 任务管理 / 番茄钟社交功能

## 7. 验收思路

如果未来把 fork 的 phase 语义正式落地，至少应满足：

1. 修改默认专注 / 休息设置时，不会打断当前 phase
2. 当前专注阶段可以安全地延长或提前进入休息
3. `提前休息 5 分钟` 在剩余时间不足 5 分钟时不可用
4. 当前休息阶段可以安全延长 1 分钟或 5 分钟
5. session 记录保存的是当前轮次最终计划休息时长，而不是旧默认值
6. 菜单栏入口在常见异常场景下仍可见或可恢复
7. 菜单栏入口保留图标，并在专注和休息阶段显示实时倒计时，且不出现明显抖动
8. `桌前休息` 可以在不展示全屏遮罩的情况下继续计时，而其他休息类型仍保持更强的打断感
