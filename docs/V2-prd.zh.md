# Rhythm V2 PRD（Fork 草案）

## 1. 文档定位

本文档用于记录我在本地 fork 上继续推进的产品方向与近期待办，不替代上游 V1 设计文档。

- `docs/V1-design.md` 继续作为上游 `main` 分支的 V1 基线与历史设计记录
- 本文档描述 fork 中已经探索过的交互差异，以及接下来可能推进的 V2 方向

换句话说：如果要了解当前上游 V1 是什么，看 `docs/V1-design.md`；如果要了解这个 fork 准备往哪里走，看本文档。

## 2. 当前基线与已探索差异

### 2.1 上游 V1 基线

当前上游 `main` 仍然以稳定可用的节奏提醒工具为目标，核心能力包括：

- 菜单栏常驻应用
- 专注 / 休息节奏配置
- 全屏休息遮罩
- `ESC` 跳过休息
- 锁屏后重置周期
- 不休息模式
- 本地 session 记录

### 2.2 Fork 已落地的行为变化

本地 fork 现在已经在上游 V1 基线之外落地了以下行为：

1. 当前专注阶段可直接调整当前 phase
   - `延长专注 5 分钟`
   - `延长专注 10 分钟`
   - `提前休息 5 分钟`
2. `提前休息 5 分钟` 只在至少还剩 5 分钟时可用
3. 当前休息阶段可直接调整当前 phase
   - `延长休息 1 分钟`
   - `延长休息 5 分钟`
   - 非阻塞 `桌前休息` 进行中时，可以 `缩短休息 5 分钟`
4. 默认设置与当前 phase 的语义被拆开
   - 修改 `专注间隔` 与 `休息时长` 时，影响下一轮周期
   - 当前 phase 的增减时长，只影响当前 phase
5. 菜单栏可见性与相关 UI 稳定性经过额外加固
   - 包括菜单栏图标可见性优化
   - 菜单栏中原本显示的 `Rhythm` 文案已改为保留图标并展示实时倒计时
   - 包括状态项被系统移除后的恢复能力
6. fork 现在已经支持中英双语界面
   - 菜单栏设置中可以切换 `中文` 和 `English`
   - 首次使用时，只有 `zh*` 系统语言默认中文，其它系统语言默认英文
   - 后续新增的用户可见功能默认应继续保持双语
7. fork 现在已经落地了更轻量的休息模型与本地 history 基线
   - 普通休息默认时长现在最高可设到 60 分钟
   - 菜单中只保留 `桌前休息` 作为显式的“继续用电脑但不工作”休息动作
   - `桌前休息` 明确是非锁屏、可继续使用电脑的休息类型，并可在最后 5 分钟提醒
   - 锁屏后的离屏时间会作为隐藏休息累计到解锁为止，然后开始新的专注周期
   - 如果用户没先锁屏、而是让 Mac 直接进入睡眠，睡眠时间也会作为隐藏休息累计；若唤醒后先看到锁屏，则会继续累计到真正解锁为止
   - 正常退出或关机后的应用关闭时间，会在下次启动时作为隐藏休息计入；异常退出则用 15 分钟 heartbeat 兜底估算，单次最多计入 12 小时
   - 本地 history 现在会把专注 / 休息 session 按周写入 Application Support 下的 JSON 目录
8. 每日总量与本地 history 现在已经拥有独立浏览入口
   - 菜单中保留紧凑的今日专注 / 休息内联总量摘要，并提供进入数据概览窗口的入口
   - 独立的数据概览窗口展示 今日、最近 7 天、最近 30 天、全部历史 汇总，并用紧凑方式呈现范围总量
   - 数据概览窗口按统计日逐天浏览 session，默认不把隐藏休息混进列表，并支持把 今日 / 最近 7 天 / 最近 30 天 / 全部历史 / 当前所选统计日 导出为 CSV 或 JSON
   - 总量统计、图表 bucket 与 history 分组共用同一个可配置的日切换点（`00:00`-`23:00`）

## 3. V2 想解决的问题

V1 已经能完成“提醒休息”这件事，但 fork 想进一步解决的是：当用户正在使用应用时，是否能更自然地微调当前节奏，而不必被迫重置整个周期。

V2 的核心目标：

1. 让“默认节奏”和“当前 phase 调整”成为两套清晰、不互相打断的规则
2. 让常见临时决策可以在菜单栏或遮罩上直接完成，而不是通过重置计时曲线来实现
3. 把双语界面作为后续功能的默认基线，并在保持菜单轻量的前提下，把更完整的 history 放进独立窗口

## 4. V2 产品方向

### 4.1 Phase 级临时控制

保留并整理 fork 中已经出现的 phase 控制模型：

- 专注阶段支持：
  - 提前休息 5 分钟
  - 延长专注 5 分钟
  - 延长专注 10 分钟
  - 当前专注进入最后 5 分钟时，在系统允许通知的情况下发送一次带默认系统提示音的提醒
- 休息阶段支持：
  - 延长休息 1 分钟
  - 延长休息 5 分钟
  - 当前 `桌前休息` 至少还剩 5 分钟时，可以缩短 5 分钟
  - 把正在展示全屏遮罩的普通休息切换成 `桌前休息`，并保留同一个剩余倒计时
  - 当前 `桌前休息` 进入最后 5 分钟时，在系统允许通知的情况下发送一次带默认系统提示音的提醒

该模型的设计原则：

- 默认设置代表“下一轮应该怎样开始”
- phase 控制代表“这一轮现在怎么调整”
- 两类操作不能互相覆盖语义

### 4.2 设置生效规则

V2 倾向采用以下规则作为正式语义：

1. 用户修改 `专注间隔` 时，不重置当前专注阶段
2. 用户修改 `休息时长` 时，不改变已经开始的休息阶段
3. 新设置在下一轮开始时生效
4. 手动延长 / 缩短 phase 只作用于当前轮次

这样做的好处是：

- 避免用户在调默认值时意外打断当前节奏
- 避免“修改设置”和“临时补时”混成同一种操作
- 为后续引入更多快捷控制打下统一规则

### 4.3 菜单栏稳定性

fork 会继续把菜单栏存在感和可恢复性视为基础体验，而不是额外 polish。

短期内默认认为以下目标成立：

- 菜单栏图标在浅色/深色环境下都应足够清晰
- 菜单栏状态项在专注和休息阶段都应显示当前倒计时，并尽量避免宽度抖动
- 状态项不应因为系统或 UI 状态异常而悄悄消失
- 即使出现异常，也应优先恢复菜单栏入口，而不是要求用户重启应用
- 如果用户再次启动 Rhythm，包括打包版本和 Xcode / debug 版本互相重复启动，新的副本应激活已有实例并退出；主实例也应在运行期间清理后续启动的重复副本，避免两个计时器同时运行

## 5. 近期方向

以下内容构成 V2 近期路线，其中有些已经成为当前基线，有些仍然是待明确的产品方向：

### 5.1 双语界面基线

fork 现在把双语界面视为已落地的基础能力，而不是未来候选项。

当前基线包括：

- 可见应用界面同时支持中文和英文
- 菜单栏面板中提供 `中文` / `English` 语言切换
- 首次使用时采用简单规则：`zh*` 系统语言使用中文，其它系统语言使用英文
- 后续新增的用户可见功能应直接提供中英双语，而不是先做单语版本

### 5.2 休息模型基线

当前 fork 已发布的休息模型，已经从早期的“多种长休息预设”收敛为更轻量的方案：

- 普通短休息仍然保留，并且默认时长现在最高可设到 60 分钟
- `桌前休息` 是明确的“还在电脑前，但不做工作”的休息类型，适合看视频、刷动态等场景
- 普通全屏休息可以直接改成 `桌前休息`，不会重新开始休息倒计时
- 超过 5 分钟的 `桌前休息` 进入最后 5 分钟时，Rhythm 会提醒一次；`桌前休息` 结束时，Rhythm 会自动恢复专注，并在系统允许通知时提醒用户
- 真正离开工位的时间用锁屏来表示；锁定到解锁之间会被记为隐藏休息

这样做的目的是：只保留一个清晰的“可继续用电脑”的休息动作，而不是继续维护一组离屏长休息预设。

### 5.3 每日专注 / 休息总量与数据概览窗口

在不把菜单面板做成重型报表的前提下，fork 现在已经补上轻量统计和独立的数据概览窗口，帮助用户回答几个基础问题：

- 今天大概专注了多久
- 今天大概休息了多久
- 最近一周或一个月大概是什么节奏
- 这些总量具体由哪些 session 构成

当前已发布的呈现方式：

- 菜单中保留今日专注与休息内联总量，并提供打开数据概览窗口的入口
- 数据概览窗口展示 今日、最近 7 天、最近 30 天、全部历史 四个部分，并用紧凑的内联总量与遵循统计日切换点的明确日期范围来说明图表；其中 `全部历史` 使用按月聚合
- 数据概览窗口按统计日逐天浏览 session，默认不把隐藏休息直接混进列表，并支持把固定范围加上当前所选统计日导出为 CSV / JSON
- 用户可以设置统计的日切换点，而不必强制按午夜切分

### 5.4 重新审视锁屏重置

当前 shipped fork 已经不再把锁屏视为单纯的“立刻重置”。

现在的行为是：

- 锁屏时结束当前可见的专注或休息片段
- 锁定到解锁之间的时间作为隐藏休息记录
- 解锁后自动开始新的专注周期

这样更贴近“我离开工位了”的真实用法，也减少了额外长休息预设的必要性。

### 5.5 睡眠期间的隐藏休息

当前 shipped fork 也会把系统睡眠视为离开工位的隐藏休息，避免用户没有先锁屏时，睡眠时间被误算成仍在专注或休息。

现在的行为是：

- 睡眠会在入睡时结束当前可见的专注或计时休息片段
- 睡眠区间会被记录为隐藏休息
- 如果机器唤醒后直接回到桌面，隐藏睡眠休息会在唤醒时结束，并开始新的专注周期
- 如果机器唤醒后先进入锁屏界面，隐藏睡眠休息会继续累计到用户真正解锁
- 如果机器在睡眠前就已经处于锁屏状态，这段时间会继续保留为同一段隐藏锁屏休息，而不会拆成第二种隐藏休息

### 5.6 应用关闭期间的休息恢复

当前 shipped fork 也会把应用关闭期间视为隐藏休息，避免用户退出 Rhythm 或关机后，每日总量直接丢失这一段时间。

现在的行为是：

- 正常退出应用或 macOS 预期终止路径会记录精确退出时间
- 下次启动时，退出到启动之间的间隔会写入为隐藏的应用关闭休息
- 15 分钟 heartbeat 会作为异常退出兜底，用于估算强退、崩溃或断电前最后一次仍在运行的时间
- 单次应用关闭间隔最多计入 12 小时；超过上限的部分不会在 7 天汇总柱图里显示特殊空白段
- 应用关闭休息会计入今日总量与 7 天趋势，但不会出现在最近 session 列表

### 5.7 History 与导出体验

fork 现在不再把本地 history 只当作一堆原始 JSON 文件。

当前行为：

- 菜单中保留紧凑的 Today 摘要、内联总量和轻量 Recent Sessions
- 用户可以从菜单打开单例的数据概览窗口
- 数据概览窗口在一个可滚动页面里展示 今日、最近 7 天、最近 30 天、全部历史，并用紧凑内联总量与明确日期范围解释图表；`全部历史` 使用按月柱图，避免数据变多后过密
- session 浏览器一次只展示一个统计日，并在该日期内支持 `全部` / `专注` / `休息` 过滤
- 锁屏、睡眠、应用关闭产生的隐藏休息会计入总量、图表与导出，但只有在用户打开 `显示隐藏休息` 后才会进入列表
- 导出采用明确范围：今日、最近 7 天、最近 30 天、全部历史，以及当前所选统计日，可导出为 CSV 或 JSON

## 6. 非目标

当前 V2 草案仍然不准备立即引入：

- 云同步
- 多设备同步
- 复杂报表或图表系统
- 任务管理 / 番茄钟社交功能

如果之后重新评估 Apple 生态内的 companion sync，它也应该被视为独立的后续项目，而不是直接塞进当前 V2 基线。当前这轮可行性判断是：

- iPhone 和 Apple Watch 大概率需要各自的 companion app target，而不是试图把现有 macOS 菜单栏应用直接延展过去
- 如果只考虑 Apple 生态内同步，又不想先引入自建后端，私有 iCloud / CloudKit 是最可能的一方方案
- 活跃计时器的同步应建模为共享的 phase snapshot 与已完成 session 记录，而不是逐秒同步倒计时数字
- 更稳妥的第一步是只读 companion 视图；远程控制计时器和跨设备冲突处理应该放到后面再设计

Mac App Store 发布准备同样不属于当前 V2 范围。如果未来重新评估，可以先参考这份待办清单：[docs/mac-app-store-checklist.zh.md](mac-app-store-checklist.zh.md)。

## 7. 验收思路

如果未来把 fork 的 phase 语义正式落地，至少应满足：

1. 修改默认专注 / 休息设置时，不会打断当前 phase
2. 当前专注阶段可以安全地延长或提前进入休息
3. `提前休息 5 分钟` 在剩余时间不足 5 分钟时不可用
4. 当前休息阶段可以安全延长 1 分钟或 5 分钟；当前 `桌前休息` 至少还剩 5 分钟时，可以安全缩短 5 分钟
5. session 记录保存的是当前轮次最终计划休息时长，而不是旧默认值
6. 专注阶段进入最后 5 分钟时只提醒一次；如果用户延长专注并让剩余时间重新超过 5 分钟，之后再次进入阈值时可以再次提醒
7. 正在展示全屏遮罩的普通休息可以切换成 `桌前休息`，且不重置剩余倒计时
8. 菜单栏入口在常见异常场景下仍可见或可恢复
9. 菜单栏入口保留图标，并在专注和休息阶段显示实时倒计时，且不出现明显抖动
10. 重复启动，包括主实例已运行后的后续启动，不应留下两个 Rhythm 计时器同时运行
11. `桌前休息` 可以在不展示全屏遮罩的情况下继续计时
12. 超过 5 分钟的 `桌前休息` 进入最后 5 分钟时只提醒一次；如果用户延长休息并让剩余时间重新超过 5 分钟，之后再次进入阈值时可以再次提醒
13. 每日总量保持菜单级轻量呈现，而更完整的 history 与导出放进独立的数据概览窗口
14. 锁屏时间会计入休息总量，并在解锁后开始新的专注周期
15. 系统睡眠时间会计入隐藏休息，并在唤醒或解锁后开始新的专注周期
16. 应用关闭期间会通过正常退出时间或 heartbeat 兜底计入隐藏休息，且单次最多计入 12 小时
17. 隐藏休息会计入总量、趋势与导出，但默认不会直接出现在 session 列表里，除非用户显式打开
18. 导出支持 Today、最近 7 天、最近 30 天、全部历史，以及当前所选统计日，且同时提供 CSV / JSON

## 8. Open Questions

以下问题暂时保留，不在本文档中强行定案：

1. 导出未来是否需要支持自定义日期范围，而不只停留在 今日 / 最近 7 天 / 最近 30 天 / 全部历史
2. 用户自定义休息预设是否应该作为独立功能回归，而不是继续混在总量统计能力里
3. 如果之后重新评估 Apple companion sync，第一期是否应该先做 iPhone 优先的 history / 今日统计，再补一个更轻量的 Apple Watch 实时状态 companion
4. 如果未来要共享活跃计时器状态，phase 切换的所有权应该落在哪个设备上，以及多端冲突要如何解决
